= Code Blocks
:page-aliases: copy-to-clipboard.adoc

This page describes how code blocks are rendered in this UI and what behavior is available out of the box.

In AsciiDoc, code blocks are referred to as source and listing blocks.
A source block is a listing block that has a source language defined on it (for example, `javascript`).
Refer to xref:antora:asciidoc:source.adoc[source blocks] to learn more about source blocks in AsciiDoc and how they are used in Antora.

== Syntax highlighting

This UI highlights source blocks at build time using Shiki through the bundled AsciiDoc extension.
That means the rendered HTML already contains the highlighted markup when the page is generated.
No client-side syntax highlighter is required.

To enable that behavior in an Antora site, register the bundled AsciiDoc extension in the playbook:

[,yaml]
----
asciidoc:
  extensions:
    - require: ./build/extensions/asciidoc-tldr-ui.js
----

Shiki is initialized once during the build, and the converter applies highlighting when a listing block has a language.
Untyped listing blocks continue to render as plain preformatted text.

== Copy to clipboard

This UI adds a clipboard button to copyable code blocks and computes the copied value at build time.
The browser only handles the click interaction and writes the prepared value to the clipboard.

IMPORTANT: Copy to clipboard only works for a local site or if the site is hosted over HTTPS.
The clipboard API is not available on an insecure site served over plain HTTP.
In that case, the button will still render, but copying will not work.

=== Source blocks

Source blocks with a language get:

* build-time syntax highlighting
* a code header
* a copy-to-clipboard button
* a language chip with a static icon when one is available
* Shiki transformer support for annotated examples

You can try this behavior below:

[,ruby]
----
puts 'Take me to your clipboard!'
----

=== Transformer annotations

This UI enables several Shiki transformers for annotated code examples.

Annotations are configured using `code.*` block attributes so the visible code stays clean:

[source,ts]
[code.add=2,code.remove=1]
----
console.log('old')
console.log('new')
console.log('same')
----

Supported attributes include:

* `code.add="2,4"` and `code.remove="3"` for diff views
* `code.highlight="1,3-4"` for line highlighting
* `code.focus="2"` for focus mode
* `code.error="3"`, `code.warning="2"`, and `code.info="4"` for severity highlighting
* `code.word="term,other"` for word highlighting

Example:

[,asciidoc]
----
[source,json]
[code.highlight="3",code.word="hello"]
----

Additional block attributes supported by this UI:

* `code.whitespace=true` or `code.whitespace=boundary` renders visible whitespace markers
* `code.indent-guides=true` renders indent guides
* `code.indent="4"` adjusts the indent-guide width when guides are enabled

These annotations affect the rendered code, but the copied text remains clean because the source itself is not modified.

NOTE: Standard Asciidoctor callout explanations still use native inline callout markers such as `<1>`.

=== Console blocks

The UI also supports console-style examples.
A console block is a literal block that begins with `$ ` on each command line.

When such a block is copied, the visible prompts are not copied.
Instead, the UI extracts the commands at build time and joins separate commands using `&&`.

In the following example, the copied value becomes `mkdir example && cd example`:

[,console]
----
$ mkdir example
$ cd example
----

That makes tutorial-style shell examples easier to copy and paste directly into a terminal.
